# DATA_DICTIONARY — KaspiPulse

Описание всех колонок в публикуемых датасетах и аналитических объектах.

Два слоя хранения:
- **GitHub** — файлы CSV (gzip): сырые срезы `data/weekly/`, очищенные
  `data/weekly/clean/`. Имена полей = `scraper/parse.py` (`FIELDS`).
- **BigQuery** — таблица `snapshots` (загрузка очищенных срезов) и вью
  `products_timeseries` / `niche_summary` (аналитика). Раньше аналитику считали
  `pipeline/estimate.py` / `pipeline/aggregate.py` — теперь это источник в `sql/`,
  Python оставлен как референс. Состав полей вью совпадает с прежним Python-выходом.

> **Дисклеймер.** Все поля с префиксом `est_` — **оценочные** (estimated), а не
> фактические продажи/выручка. Методология — review-velocity (см. README и
> PROJECT_PLAN.md, раздел 2). Kaspi не публикует реальные продажи.

## 1. `data/weekly/YYYY-MM-DD.csv.gz` — дневной срез ниши (сырой)

Один файл = снимок ниши на дату сбора. Формат: CSV, сжатие gzip, кодировка UTF-8.
Источник колонок: `parse.py` → `FIELDS`.

| Колонка | Тип | Поле Kaspi (источник) | Описание |
|---|---|---|---|
| `product_id` | строка | `id` (или `configSku`) | Устойчивый идентификатор товара. Ключ для матчинга карточки между неделями. |
| `name` | строка | `title` (или `shortNameText`) | Название товара. Очищено от ведущих/хвостовых пробелов. |
| `brand` | строка | `brand` | Бренд. Может быть пустым, если Kaspi не вернул значение. |
| `price` | число (₸) | `unitPrice` (или `unitSalePrice`) | Цена в тенге. Kaspi обычно отдаёт цену уже со скидкой; `unitPrice` и `unitSalePrice` чаще всего равны. Приведена к числу из строки с пробелами/запятыми. |
| `rating` | число (0–5) | `rating` | Средний рейтинг товара. Может быть пустым. |
| `reviews_count` | целое | `reviewsQuantity` | Количество отзывов. **Ключевое поле** для оценки продаж (review-velocity). Пусто → 0. |
| `stock` | число | `stock` | Флаг наличия. В выдаче категории Kaspi всегда `= 1` («в наличии»), реального остатка на уровне списка нет (см. PROJECT_PLAN, раздел 4). |
| `best_merchant` | строка | `bestMerchant` | Главный (лучший) продавец товара. Общее число продавцов в выдаче недоступно — хранится только основной. |
| `category_id` | строка | `categoryId` | Идентификатор категории товара у Kaspi. |
| `position` | целое | — (вычисляется) | Сквозная позиция карточки в выдаче на момент сбора (1 = первая). Зависит от сортировки `relevance` и может меняться между срезами. |
| `url` | строка | `shopLink` | Полная ссылка на карточку: относительный `shopLink` достроен до `https://kaspi.kz/shop...`. |
| `captured_at` | дата (YYYY-MM-DD, UTC) | — (вычисляется) | Дата сбора среза в UTC. Совпадает с именем файла. |

## 2. `data/weekly/clean/YYYY-MM-DD.csv.gz` — очищенный срез + таблица BigQuery `snapshots`

Выход `scraper/clean.py`: тот же срез после дедупа SKU, фильтра диапазона цен и
пометки выбросов. **Именно этот файл ежедневно грузится в BigQuery** в таблицу
`snapshots` (партиц. по `captured_at`, декоратор `snapshots$YYYYMMDD` →
идемпотентно). Схема таблицы `snapshots` = колонки этого файла один-в-один
(`sql/01_snapshots.sql`).

Все колонки из раздела 1 плюс две служебные:

| Колонка | Тип | Описание |
|---|---|---|
| `seen_before` | булево | `True`, если этот `product_id` встречался в предыдущем срезе (стабильность ID между днями). Для первого появления — `False`. |
| `outlier` | булево | `True`, если значение помечено как выброс по IQR-правилу (`Q3 + 3·IQR`). Строка **не удаляется** — флаг оставлен для прозрачности; потребитель сам решает, фильтровать ли. |

Лог очистки за каждый день — `data/_logs/cleaning_report_YYYY-MM-DD.csv`
(что и почему отброшено/помечено на каждой стадии).

## 3. BigQuery вью `products_timeseries` — временной ряд с velocity

Аналитический слой поверх таблицы `snapshots`. Для каждого товара считает прирост
отзывов относительно предыдущего среза и оценку продаж. Источник: `sql/02_products_timeseries.sql`
(точный перенос `pipeline/estimate.py`). Есть вариант `products_timeseries_7d`
(`sql/04_…`) — та же методология, но база сравнения ≈7 дней назад (сглаживает
дневной шум при ежедневном кроне).

Колонки `product_id`, `name`, `brand`, `price`, `rating`, `reviews_count`,
`best_merchant`, `category_id`, `position`, `url`, `captured_at` — те же, что в
срезе (раздел 1). Колонки `stock` во вью нет.

Дополнительные (расчётные) колонки:

| Колонка | Тип | Описание |
|---|---|---|
| `delta_reviews` | целое | Прирост числа отзывов относительно **предыдущего среза этого же товара**: `reviews_count(t) − reviews_count(t−1)`. Для первого появления товара — пусто (нет предыдущего среза). При сбросе счётчика (стало меньше) принудительно `0` (см. `review_reset`). |
| `days_diff` | целое | Число дней между текущим и предыдущим срезом данного товара. Для первого появления — пусто. |
| `delta_reviews_weekly` | число | `delta_reviews`, нормированный к недельной скорости: `delta_reviews / (days_diff / 7)`. Если между срезами прошло не 7 дней — масштабируется, чтобы пропуск недели не завышал оценку. |
| `est_sales` | число (оценка) | **Оценочные** недельные продажи: `delta_reviews_weekly / REVIEW_RATE` (`REVIEW_RATE = 0.02`). Не фактические продажи. |
| `est_revenue` | число (₸, оценка) | **Оценочная** недельная выручка: `est_sales × price`. Не фактическая выручка. |
| `review_reset` | булево | `True`, если число отзывов уменьшилось по сравнению с прошлым срезом (сброс/пересчёт счётчика Kaspi). В этом случае `delta_reviews` обнуляется во избежание отрицательного прироста. |

## 4. BigQuery вью `niche_summary` — сводка по нише (одна строка на дату среза)

Агрегация поверх `products_timeseries`: `GROUP BY captured_at`. Источник:
`sql/03_niche_summary.sql` (перенос `pipeline/aggregate.py`).

| Колонка | Тип | Описание |
|---|---|---|
| `captured_at` | дата | Дата среза. |
| `niche` | строка | Название ниши (`Товары для дома`). |
| `niche_slug` | строка | Слаг ниши (`home`). |
| `total_products` | целое | Число активных SKU в срезе (`COUNT(DISTINCT product_id)`). |
| `total_est_revenue` | число (₸, оценка) | Суммарная оценочная выручка ниши: `SUM(est_revenue)`. |
| `avg_price` | число (₸) | Средняя цена по срезу. |
| `median_price` | число (₸) | Медианная цена (`PERCENTILE_CONT(price, 0.5)`). |
| `top10_revenue_share` | число (0–1) | Доля оценочной выручки топ-10 товаров (индекс концентрации). Пусто, если выручки ещё нет (первый срез). |
| `composite_score` | число (0–1) | «Горячесть» ниши: взвешенная min-max композиция 4 сигналов — `delta_reviews_weekly` (0.40), `reviews_count` (0.20), `5 − rating` (0.20), `1/price` (0.20), усреднённая по товарам. При равных значениях сигнала внутри среза — нейтраль 0.5. |

## Примечания

- **Первый срез товара.** В строках, где товар встречается впервые,
  `delta_reviews`, `days_diff`, `delta_reviews_weekly`, `est_sales`,
  `est_revenue` пусты — динамику не из чего считать.
- **`est_*` — оценки, не факты.** Используйте для сравнения товаров/ниш и
  трендов, а не как точные продажи. Коэффициент `REVIEW_RATE` подлежит
  калибровке на накопленных данных.
- **Стабильность `position`.** Сортировка `relevance` у Kaspi подмешивает новые
  карточки — позиция одного товара может колебаться между неделями.
