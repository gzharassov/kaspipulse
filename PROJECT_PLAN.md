## 1. Что это и зачем

**KaspiPulse** — некоммерческий open-data проект Academica. Ежедневно собирает
публичные данные по одной товарной нише Kaspi.kz, оценивает спрос по косвенным
метрикам (методология review-velocity, как у Jungle Scout / Helium 10 для Amazon),
публикует чистые датасеты и дашборд. Архитектура: GitHub (сырьё/clean) → BigQuery
(аналитика в SQL) → Looker Studio (дашборды).

**Зачем:**
- Дать студентам Academica реальный аналитический опыт и публичное портфолио.
- Каждый студент в резюме пишет узнаваемое: *«Анализ данных маркетплейса Kaspi.kz:
  сбор и обработка датасета, оценка спроса по review-velocity, построение дашборда»*.
- Проект самоподдерживается бесплатно (GitHub Actions), без серверов.

**Чем НЕ является:** не клон коммерческого RedStat и не конкурент. Скоуп узкий,
частота низкая (раз в неделю), цель — обучение и портфолио.

### Юридическая чистота
- «Kaspi» используется **только как объект анализа**. Проект не аффилирован с АО «Kaspi.kz».
- Лицензия данных: **CC BY-NC 4.0**.
- Собираются только публичные данные витрины. Не трогаем персональные данные,
  не обходим капчу/авторизацию, не создаём избыточную нагрузку.

## 2. Методология оценки продаж

Ни Amazon, ни Kaspi не отдают реальные продажи. Оцениваем косвенно.

### Основной метод: Review velocity с нормировкой по дням

Прирост числа отзывов между срезами ≈ прокси продаж.

```
delta_reviews        = reviews(t) − reviews(t−1)
delta_reviews_weekly = delta_reviews × 7 / days_between_snapshots
est_sales            ≈ delta_reviews_weekly / REVIEW_RATE
est_revenue          ≈ est_sales × price
```

**Нормировка по дням — критична.** Если между срезами прошло 14 дней (пропустили
неделю), без нормировки `est_sales` будет завышен ровно в 2 раза. Реализовано в
BQ-вью `products_timeseries` (`sql/02`); эталон — `pipeline/estimate.py` (референс).

> **Где считается аналитика.** velocity, `est_sales`, `est_revenue`, сводка по
> нише и `composite_score` живут как **BigQuery SQL-вью** (`sql/`) — единственный
> источник истины. Python-модули `estimate.py`/`aggregate.py` оставлены как
> проверенный референс методологии и для парити-чека, но в ежедневном пайплайне
> не вызываются. Методология при переносе не менялась.

### Калибровка `REVIEW_RATE`

Стартовое значение — **2%** (по аналогии с Amazon). Для Казахстана реальная цифра
почти наверняка выше, потому что **Kaspi платит бонусы за отзывы** — стимулирует
писать чаще. Калибровка коэффициента на накопленных данных — отдельная
аналитическая задача (после 4+ недель сбора).

### Дополнительные методы (заложены, частично доступны)

- **Δостатков (Метод 2).** В выдаче категории Kaspi отдаёт `stock = 1` как флаг
  «в наличии», без реального остатка. На уровне списка метод недоступен; реальный
  остаток виден только на странице товара, куда мы не ходим из соображений нагрузки.
- **Композитный скоринг ниши (Метод 3).** Связка цены, абсолютных отзывов, темпа
  их прироста и числа продавцов → ранжирование «горячих» ниш с низкой
  конкуренцией. Реализован в BQ-вью `niche_summary` (`sql/03`); эталон —
  `pipeline/aggregate.py` (референс).

### Дисклеймер в данных
Все продажи и выручка помечаются как **`est_*`** (estimated) — оценочные,
не фактические. Указывается в README и DATA_DICTIONARY.

## 3. Подтверждённые технические факты Kaspi

Всё проверено через DevTools на боевых запросах.

**Эндпоинт выдачи:**
```
GET https://kaspi.kz/yml/product-view/pl/results
Параметры: page, q, sort, ui, i, c, requestId
```

**Структура ответа JSON:** товары лежат во вложенности `data → data → [список]`.

**Параметр `q`** задаёт категорию и зону доставки:
```
:category:<КОД>:availableInZones:<ЗОНА>
```

**Города и зоны** (все подтверждены живыми запросами):

| Город     | `c` (city_id) | `availableInZones` |
|-----------|---------------|--------------------|
| Астана    | 710000000     | Magnum_ZONE5       |
| Алматы    | 750000000     | Magnum_ZONE1       |
| Шымкент   | 511010000     | 511010000          |

Kaspi использует две системы зон: именованные `Magnum_ZONE*` для городов с
Magnum-партнёрством (Алматы, Астана) и `zone = city_id` для остальных.

**Поля карточки товара** (то, что реально приходит):
`id`, `configSku`, `title`, `brand`, `unitPrice`, `unitSalePrice`, `priceFormatted`,
`rating`, `reviewsQuantity`, `stock` (≡ 1 в списке), `bestMerchant`, `majorMerchants` ([]),
`categoryId`, `shopLink`, `previewImages`, `deliveryDuration`, `deliveryZones`.

## 4. Известные ограничения данных (фиксируем честно)

- **`n_sellers` недоступно в списке.** Kaspi отдаёт только `bestMerchant` (главный
  продавец), общего числа продавцов на товар в выдаче категории нет. Получить
  можно только отдельным запросом `sellers?sku=...` на каждый товар — это
  увеличит нагрузку в 50–100 раз, неэтично. Решение: храним `best_merchant`,
  в README честно описано ограничение.
- **`stock` в списке ≡ 1.** Реальный остаток есть только на странице самого
  товара. Метод дельты остатков на уровне категории не работает.
- **Ротация выдачи.** Kaspi при одинаковой сортировке `relevance` подмешивает
  новые карточки, состав топа от среза к срезу может слегка меняться. Аргумент в
  пользу сужения до конкретной подкатегории, где выдача стабильнее.
- **Цена в JSON.** Поля `unitPrice` и `unitSalePrice` обычно равны (Kaspi уже
  отдаёт цену со скидкой). Хранится `unitPrice`, выручка считается по ней.

## 7. Пайплайн очистки (7 стадий — `clean.py`, ✅ реализовано)

`clean.py` забирает свежий CSV из `data/weekly/`, прогоняет 7 стадий и пишет
`data/weekly/clean/<дата>.csv.gz` (именно он грузится в BigQuery) + лог:

1. **Null/мусор** — отсев товаров без обязательных полей.
2. **Дедупликация SKU** — один `product_id` = одна строка. Минимальная цена,
   склейка `best_merchant` через запятую.
3. **Диапазоны цен** — отсечь нереальные. **Осторожно:** на Kaspi есть
   легитимные дешёвые товары (салфетки 19 ₸, расходники). Пороги широкие:
   `5 ₸ ≤ price ≤ 5 000 000 ₸`. Логировать всё, что отсеяно.
4. **Стабильность ID** — товар матчится между днями по `product_id` (флаг
   `seen_before`).
5. **Сброс отзывов** — если стало меньше отзывов, чем в прошлом срезе → `Δ = 0`
   и флаг `review_reset` (оставлено в слое velocity — `sql/02` / `estimate.py`).
6. **Выбросы продаж** — аномально большой `Δreviews`/значение помечается
   `outlier`, но не удаляется (для прозрачности).
7. **Логирование** — что отброшено и почему → `data/_logs/cleaning_report_<дата>.csv`.

## 8. Композитный скоринг ниши (BQ-вью `niche_summary`, ✅ реализовано)

BQ-вью `niche_summary` (`sql/03`, перенос `aggregate.py`) читает
`products_timeseries` и для каждой даты среза считает по нише:

- `total_est_revenue` — суммарная оценочная выручка.
- `total_products` — число активных SKU.
- `avg_price`, `median_price`.
- `top10_revenue_share` — доля выручки топ-10 товаров (индекс концентрации).
- `composite_score` — «горячесть» ниши, нормированная композиция:
  `delta_reviews_weekly` (вес 0.40) + `reviews_count` (0.20)
  + `(5 − rating)` (0.20) + обратная цена `1/price` (0.20).

Веса и min-max нормировка (нейтраль 0.5 при равных значениях) — те же, что в
референсе `aggregate.py`.

## 9. Этичный скрейпинг (обязательно)

- Реальный User-Agent, случайные паузы 2–5 сек, экспоненциальные ретраи.
- Уважение `robots.txt`. Без обхода капчи и авторизации.
- Узкая категория + один проход в день = десятки запросов, минимальная нагрузка.
- Только публичные данные витрины, никаких персональных данных.
- Если Kaspi меняет структуру — скрейпер падает с понятной ошибкой; студент
  чинит селекторы через DevTools. Это часть учебного процесса.
