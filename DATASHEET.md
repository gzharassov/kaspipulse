# DATASHEET — KaspiPulse

Описание датасета по стандарту **Datasheets for Datasets** (Gebru et al., 2021).
Документ фиксирует происхождение, состав, способ сбора, ограничения и условия
использования данных KaspiPulse.

- **Датасет:** KaspiPulse — ежедневные срезы одной товарной ниши Kaspi.kz.
- **Версия документа:** 2026-05-30.
- **Лицензия данных:** CC BY-NC 4.0 · **Лицензия кода:** MIT (см. `LICENSE`).
- **Сопутствующая документация:** `README.md`, `PROJECT_PLAN.md`, `DATA_DICTIONARY.md`.

---

## 1. Motivation (Мотивация)

**Для чего создан датасет?**
KaspiPulse — некоммерческий учебный open-data проект Academica. Цель — дать
студентам реальный аналитический опыт и публичное портфолио: сбор и обработка
датасета маркетплейса, оценка спроса по косвенным метрикам (review-velocity),
построение дашборда. Датасет позволяет изучать динамику цен, отзывов и
оценочного спроса в одной нише Kaspi.kz во времени.

**Кто создал и кто финансирует?**
Проект Academica Courses. Самоподдерживается бесплатно на GitHub Actions, без
серверов и внешнего финансирования. Проект **не аффилирован** с АО «Kaspi.kz»;
«Kaspi» фигурирует исключительно как объект анализа.

**Какую задачу решает?**
Оценку относительного спроса и «горячести» товаров в нише при отсутствии
публичных данных о реальных продажах — по приросту числа отзывов между срезами.

## 2. Composition (Состав)

**Что представляют собой записи?**
Одна запись = один товар (SKU) в выдаче категории Kaspi.kz на дату сбора.
Единицы данных — публичные карточки товаров витрины.

**Какие данные содержит?**
Полное описание всех колонок — в `DATA_DICTIONARY.md`. Кратко:
- идентификатор товара, название, бренд, цена, рейтинг, число отзывов,
  флаг наличия, главный продавец, категория, позиция в выдаче, ссылка, дата среза
  (`data/weekly/*.csv.gz`);
- плюс расчётная динамика: прирост отзывов, интервал в днях, недельная скорость,
  **оценочные** продажи и выручка, флаг сброса счётчика
  (`data/derived/products_timeseries.csv.gz`).

**Сколько записей?**
Зависит от ниши и числа накопленных срезов. Пилот: ниша «Товары для дома»
(`:category:Home`), город Астана; локальный тест собирал 24+ товара за срез.
Объём растёт на один недельный срез в неделю.

**Это выборка или вся популяция?**
Выборка верхней части выдачи категории при сортировке `relevance`
(до `MAX_PAGES = 30` страниц по ~12 товаров). Это не вся категория Kaspi, а
наиболее релевантный/популярный её срез.

**Есть ли персональные данные?**
Нет. Собираются только публичные данные витрины: характеристики товаров и
агрегированные метрики. Имена продавцов (`best_merchant`) — это публичные
названия магазинов/мерчантов, не персональные данные покупателей. Тексты
отзывов, данные пользователей не собираются.

**Есть ли конфиденциальные/чувствительные данные?**
Нет.

**Метки/целевые значения.**
«Истинных» продаж в данных нет — Kaspi их не публикует. Поля `est_sales` и
`est_revenue` являются **оценками** (proxy), а не размеченной истиной.

## 3. Collection Process (Процесс сбора)

**Как собираются данные?**
Автоматически, GitHub Actions по расписанию (cron, ежедневно 04:00 UTC) и по
ручному запуску. Конвейер:
`scraper/fetch.py` → сырой JSON → `scraper/parse.py` → срез
`data/weekly/<дата>.csv.gz` → `scraper/clean.py` → очищенный
`data/weekly/clean/<дата>.csv.gz` → коммит в GitHub → `bq load` в BigQuery
(таблица `snapshots`). Аналитика (velocity, оценка продаж, сводка по нише)
считается уже в BigQuery SQL (`sql/`), а не в Python.

**Источник.**
Публичный JSON-эндпоинт выдачи Kaspi:
`GET https://kaspi.kz/yml/product-view/pl/results` (параметры `page, q, sort, ui,
i, c, requestId`). Товары лежат во вложенности `data → data → [список]`. HTML не
парсится; страницы отдельных товаров не запрашиваются.

**Частота.**
Один проход в день. Низкая интенсивность сбора (десятки запросов) выбрана
сознательно — минимальная нагрузка на сервис.

**Этика и вежливость сбора.**
- Реалистичный User-Agent, случайные паузы 2–5 сек между страницами,
  экспоненциальные ретраи.
- Уважение `robots.txt`. Без обхода капчи и авторизации.
- Узкая категория + один проход в день = десятки запросов за сбор.
- Только публичные данные витрины, никаких персональных данных покупателей.

**Период сбора.**
Старт — дата первого автозапуска workflow; далее накопление по неделям.

## 4. Preprocessing / Cleaning / Labeling (Предобработка и очистка)

**Что делается с сырыми данными?**
1. `parse.py` — JSON → плоская таблица; приведение чисел (удаление пробелов,
   ` `, замена запятой на точку); базовый фильтр валидности (товар без
   `product_id` или `price` отбрасывается); достройка относительной ссылки до
   полного URL.
2. `clean.py` — отдельный модуль очистки (7 стадий): отсев мусора, дедупликация
   по `product_id` (минимальная цена, склейка `best_merchant`), отсечение
   нереальных цен по **широким** порогам `5 ₸ ≤ price ≤ 5 000 000 ₸` (дешёвые
   расходники легитимны), пометка выбросов прироста без удаления, логирование
   отброшенного в `data/_logs/cleaning_report_*.csv`.
3. **Аналитика — в BigQuery SQL** (`sql/`), не в Python. Очищенный срез грузится
   в таблицу `snapshots`; вью `products_timeseries` считает `delta_reviews` с
   **нормировкой по дням**, защиту от сброса счётчика отзывов (`review_reset`),
   `est_sales` и `est_revenue`; вью `niche_summary` — сводку по нише и
   `composite_score`. Python-модули `pipeline/estimate.py` и
   `pipeline/aggregate.py` сохранены как референс методологии (парити-чек), но в
   ежедневном пайплайне не вызываются.

**Сохраняются ли сырые данные?**
Сырой JSON (`data/raw/`) **не публикуется** (в `.gitignore`) — он восстановим
повторным сбором. Публикуются сырые и очищенные срезы (GitHub); аналитические
таблицы живут как вью поверх `snapshots` в BigQuery.

**Методология оценки.**
`est_sales ≈ (Δreviews × 7 / days_diff) / REVIEW_RATE`, где `REVIEW_RATE = 0.02`
(стартовое значение по аналогии с Amazon; для Казахстана вероятно выше из-за
бонусов Kaspi за отзывы). Коэффициент подлежит калибровке на накопленных данных.

## 5. Известные ограничения (по PROJECT_PLAN, раздел 4)

Честное перечисление того, что данные **не** содержат и почему:

- **`n_sellers` (общее число продавцов) недоступно.** В выдаче категории Kaspi
  отдаёт только `bestMerchant`. Получить полное число продавцов можно лишь
  отдельным запросом `sellers?sku=...` на каждый товар — это увеличило бы
  нагрузку в 50–100 раз, что неэтично. Хранится только `best_merchant`; полной
  конкурентной картины по числу продавцов нет.
- **`stock` в списке ≡ 1.** Это флаг «в наличии», а не реальный остаток.
  Реальный остаток виден только на странице товара, куда проект не ходит. Метод
  дельты остатков на уровне категории **не применим**.
- **Ротация выдачи.** При сортировке `relevance` Kaspi подмешивает новые
  карточки, поэтому состав топа и `position` могут меняться от среза к срезу.
  Это аргумент за сужение до конкретной подкатегории, где выдача стабильнее.
- **Цена в JSON.** `unitPrice` и `unitSalePrice` обычно равны (Kaspi уже отдаёт
  цену со скидкой). Хранится `unitPrice`; выручка считается по нему.
- **Оценочность продаж.** `est_sales`/`est_revenue` — косвенные оценки через
  прирост отзывов, не фактические продажи. Точность зависит от `REVIEW_RATE`.

## 6. Uses (Использование)

**Для чего подходит датасет?**
- Учебная аналитика и портфолио: динамика цен, отзывов, оценочного спроса.
- Сравнение относительной «горячести» товаров и подниш во времени.
- Композитный скоринг ниши (BQ-вью `niche_summary`).
- Практика data engineering / BI / визуализации (дашборд).

**Для чего НЕ подходит / на что обратить внимание.**
- Не использовать `est_*` как фактические продажи/выручку — это оценки.
- Не делать выводов о полной конкуренции по числу продавцов (есть только
  `best_merchant`).
- Учитывать ротацию выдачи и нестабильность `position` при анализе трендов.
- Коэффициент `REVIEW_RATE` не откалиброван — абсолютные значения оценок условны.

**Запрещённое использование.**
Коммерческое использование данных запрещено лицензией CC BY-NC 4.0.

## 7. Distribution (Распространение)

**Как распространяется?**
Два канала:
- **GitHub** `gzharassov/kaspipulse` — публичный архив сырых и очищенных срезов
  (CSV в gzip, читаются напрямую `pandas.read_csv`). Источник истины для сырья.
- **Google BigQuery** — аналитический слой: таблица `snapshots` (загрузка
  очищенных срезов) и вью `products_timeseries` / `niche_summary`. Отсюда данные
  тянет **Looker Studio** для дашбордов. DDL и вью — в каталоге `sql/`.

## 8. Maintenance (Сопровождение)

**Кто сопровождает?**
Команда проекта Academica (Academica Courses). Контакт — через GitHub-репозиторий
(issues / pull requests).

**Как обновляется?**
Автоматически: новый срез добавляется коммитом бота ежедневно и тем же прогоном
грузится в BigQuery (`snapshots`). Временной ряд (`products_timeseries`) и сводка
по нише (`niche_summary`) — это BQ-вью, пересчитываются на лету при запросе.

**Версионирование.**
История изменений — через git-коммиты. Каждый срез датирован (`captured_at` и
имя файла).

**План развития.**
Калибровка `REVIEW_RATE` на 3–4+ срезах, дашборд в Looker Studio, сужение до
подкатегории, далее — вторая ниша/город. См. дорожную карту в PROJECT_PLAN.md.

**Что при поломке сбора?**
Если Kaspi меняет структуру ответа — скрейпер падает с понятной ошибкой; селекторы
чинятся через DevTools. Это часть учебного процесса.
